# 音乐版权合同审查助手（乙方视角）

> 从乙方立场审查音乐版权相关服务合同，基于中国现行法律，采用场景化分流审查模式，根据合同实际类型自动加载对应的审查模块。

---

## 功能概述

本工具是一个面向律师/法务的音乐版权合同审查辅助指令集，专精音乐版权行业，核心特性：

- **乙方视角锁定**：所有审查建议始终倾向于保护乙方利益，不切换立场
- **5 大业务场景覆盖**：法定许可代理(A1)、授权转授权(A2)、线下店铺音乐(B)、词曲著作权对外授权(C)、品牌歌曲定制制作(D)
- **智能场景识别**：根据合同内容自动匹配场景，附带决策树辅助区分易混淆场景
- **双审查模式**：修改+批注（模式A）/ 仅批注（模式B），审查前必须确认
- **风险分级处理**：🔴高风险 / 🟡中风险条款才修改或补充；🟢低风险不改；💼商务条款只批注
- **法律依据可溯源**：所有法律引用来自内置法律依据池，禁止编造
- **六维自检机制**：立场、法律依据、场景一致性、审阅模式、审阅稿核对、质量——审查质量有保障

## 快速开始

### 触发方式

将合同文件或审查需求提供给任意 AI Agent 即可触发，例如：

- "帮我审查这份音乐版权代理合同"
- "这份背景音乐服务合同有什么风险？"
- "审查一下品牌歌曲定制制作合同"

> **适用平台**：本工具以结构化指令（SKILL.md + 模块文件）形式交付，任何能读取 Markdown 指令的 AI Agent 均可加载使用，不限于特定平台。

### 环境要求

生成 Word 审阅稿/批注稿（模式A 或 模式B）时，需要调用 **WPS 的 COM 接口**（`Kwps.Application`）来操作修订模式和批注。因此运行环境需满足：

- **操作系统**：Windows
- **WPS Office**：已安装且已激活（需支持 COM 自动化）
- **Python 依赖**：`pywin32`（用于调用 WPS COM 接口）
- **权限**：Agent 需有本地执行 Python 脚本和启动 WPS 进程的权限

> 若仅输出审查摘要报告（不生成 Word 审阅稿），则无需 WPS 环境。

## 审查流程

```
1. 加载基础模块 → 2. 确认审查模式 → 3. 识别业务场景 → 4. 加载场景模块 → 5. 执行审查 → 6. 自检 → 7. 输出结果
```

### Step 1：加载基础模块

始终加载以下四个模块：

| 模块 | 文件 | 用途 |
|------|------|------|
| 法律依据池 | `modules/_legal-basis.md` | 所有法律引用的唯一来源（防幻觉） |
| 输出格式规范 | `modules/_output-format.md` | 统一审查报告与审阅稿格式 |
| 通用审查清单 | `modules/_common.md` | 所有合同共通的审查内容 |
| 自检机制 | `modules/_self-check.md` | 六维自检，确保审查质量 |

### Step 2：审查模式确认（必问）

审查开始前，必须先确认审查模式：

| 模式 | 名称 | 正文处理 | 批注处理 | 输出物 |
|------|------|---------|---------|--------|
| **模式A** | 修改+批注 | 🔴🟡条款用审阅模式修改；💼条款不修改 | 🔴🟡添加修改理由批注；💼添加风险提示批注 | 审查摘要报告 + Word审阅稿 |
| **模式B** | 仅批注 | 正文不做任何修改 | 所有🔴🟡💼条款均添加风险提示批注 | 审查摘要报告 + Word批注稿 |

> 若用户未明确选择，默认按**模式B**（仅批注）执行。

### Step 3：场景识别

根据合同内容匹配场景对照表：

| 代号 | 服务类型 | 乙方角色 | 授权链中位置 | 触发关键词 |
|------|---------|---------|-------------|-----------|
| **A1** | 法定许可代理（纯代理） | 代理人（非权利人） | ❌ 不在链上 | 法定许可、代办、协助申请、MCSC/CAVCA代理 |
| **A2** | 授权转授权服务 | 中间被授权方+转授权人 | ✅ 在链中间 | 转授权、再授权、版权采购、上游对接 |
| **B** | 线下店铺音乐服务 | 权利方/被许可方 | ✅ 权利端 | 门店、店铺、背景音乐、歌单、音乐服务系统 |
| **C** | 词曲著作权代理对外授权 | 被授权方（采购方） | ✅ 下游端 | 词曲代理、活动授权、独家/非独家、改编授权 |
| **D** | 品牌歌曲/音乐定制制作 | 受托方（制作方/创作方） | ✅ 创作端 | 品牌歌曲、定制歌曲、音乐制作、创作、编曲 |

> **场景识别不明确时**，工具会主动追问关键问题，不会强行归类。

### Step 4：加载场景模块

根据识别结果加载 `scenarios/scenario-[代号].md`，每个场景模块包含该场景下的专项审查要点。

### Step 5：执行审查

1. 场景专项审查（场景模块）
2. 通用条款审查（`_common.md`）
3. 风险等级标注与建议
4. **先输出审查摘要报告**，确保所有问题全面记录
5. **再根据报告逐项生成审阅稿/批注稿**，以报告中的"修改建议汇总"为清单逐条核对

### Step 6：自检

按 `_self-check.md` 进行六项自检，未通过则修正后重新自检。

### Step 7：输出结果

双输出模式，先报告后审阅稿：

- **审查摘要报告**：包含合同基本信息、场景识别、风险概览、审查详情、修改建议汇总、谈判策略提示、自检声明
- **Word 审阅稿/批注稿**：审阅人名称统一为 `legal assistant`，修订模式修改或批注风险提示

## 场景识别决策树

### 最易混淆：A1 vs A2

**核心区分：乙方是否在授权链条上**

```
乙方"取得""被授予"音乐使用权 → 再授权给甲方 → 场景 A2（转授权）
授权链：著作权人 → 乙方 → 甲方（乙方在中间）

乙方"协助""代理""代办"甲方取得授权 → 场景 A1（纯代理）
授权链：著作权人 → 甲方（乙方不在链上）
```

### 易混淆：B vs C

**核心区分：乙方是"给出权利"还是"获取权利"**

```
乙方提供音乐服务/授权给甲方使用 → 场景 B（线下店铺音乐服务）
授权方向：乙方 → 甲方

乙方购买/获取甲方音乐作品使用权 → 场景 C（词曲著作权对外授权）
授权方向：甲方 → 乙方
```

### 新场景区分：D vs B/C

**核心区分：乙方是否在为甲方创作新作品**

```
乙方"创作""制作""编曲"新作品 → 场景 D（品牌歌曲定制制作）
乙方提供已有音乐/授权 → 场景 B
乙方向甲方购买授权 → 场景 C
```

## 各场景立场与核心策略

| 场景 | 乙方立场 | 核心策略 |
|------|---------|---------|
| A1 法定许可代理 | 服务代理人 | **撇清责任**：我只是中介，不替版权效力背书 |
| A2 授权转授权 | 中间被授权方 | **平衡责任**：对上游负责，对下游约束，自身设上限 |
| B 线下店铺音乐 | 权利方/服务方 | **主张权利**：严控甲方使用范围，保护知识产权 |
| C 词曲著作权对外授权 | 被授权方/采购方 | **确保获权**：获得有效授权，转嫁权利瑕疵风险 |
| D 品牌歌曲定制 | 受托方/制作方 | **保护创作**：限制无限修改，确保收款，保留署名权 |

## 各场景核心审查要点一览

### A1 法定许可代理（纯代理模式）

| 序号 | 必查条款 | 风险等级 | 审查要点 |
|------|---------|---------|---------|
| A1-1 | 乙方责任边界 | 🔴 | 明确乙方仅为代理人，不对授权结果担保 |
| A1-2 | 授权结果不保证 | 🔴 | 乙方不保证法定许可/授权申请成功 |
| A1-3 | 甲方配合义务 | 🟡 | 甲方需提供申请所需资料/条件 |
| A1-4 | 服务费与付款 | 🔴 | 服务费标准、付款节点、与授权结果脱钩 |
| A1-5 | 违约责任限定 | 🔴 | 乙方违约责任以服务费为限，排除间接损失 |
| A1-6 | 保密条款 | 🟡 | 甲方资料保密、保密期限合理 |

### A2 授权转授权服务（中间方模式）

| 序号 | 必查条款 | 风险等级 | 审查要点 |
|------|---------|---------|---------|
| A2-1 | 上游授权有效性 | 🔴 | 乙方是否已取得上游授权、授权范围是否覆盖 |
| A2-2 | 转授权权限 | 🔴 | 乙方是否有权转授权、转授权范围限制 |
| A2-3 | 授权范围精准化 | 🔴 | 权利项、地域、期限、使用方式逐项列明 |
| A2-4 | "未明示即未授权"兜底 | 🔴 | 凡未明示授予之权利均不得行使 |
| A2-5 | 甲方使用限制 | 🟡 | 使用范围、禁止转授权、再授权限制 |
| A2-6 | 赔偿责任上限 | 🔴 | 乙方赔偿责任以授权费/合同金额为限 |
| A2-7 | 上游授权终止的处理 | 🔴 | 上游授权终止时乙方的通知义务和免责 |

### B 线下店铺音乐服务（权利方/服务方模式）

| 序号 | 必查条款 | 风险等级 | 审查要点 |
|------|---------|---------|---------|
| B-1 | 服务范围与内容 | 🔴 | 歌单/系统/更新频率等明确 |
| B-2 | 授权范围限定 | 🔴 | 仅限店铺内背景音乐播放，排除其他使用 |
| B-3 | 甲方使用限制 | 🔴 | 禁止翻录、下载、二次传播、公开演出 |
| B-4 | 服务费与付款 | 🔴 | 按门店数/面积/月收费，付款节点清晰 |
| B-5 | 知识产权保护 | 🔴 | 乙方保留所有IP，甲方不得侵犯 |
| B-6 | 违约责任 | 🟡 | 甲方超范围使用的违约金、乙方服务中断责任 |

### C 词曲著作权代理对外授权（采购方模式）

| 序号 | 必查条款 | 风险等级 | 审查要点 |
|------|---------|---------|---------|
| C-1 | 授权权利项 | 🔴 | 明确授权的权利种类（表演权/复制权/改编权等） |
| C-2 | 授权范围 | 🔴 | 地域、期限、使用方式、独家/非独家 |
| C-3 | 授权有效性保证 | 🔴 | 甲方保证其为合法权利人或有权授权 |
| C-4 | 权利瑕疵担保 | 🔴 | 第三方主张权利时甲方承担责任 |
| C-5 | 使用限制与禁止 | 🟡 | 禁止转授权、使用范围限制 |
| C-6 | 直播回放/二次使用 | 🟡 | 直播回放权利、下游平台二次使用授权 |
| C-7 | 付款与授权费 | 🔴 | 授权费标准、付款条件、发票 |

### D 品牌歌曲/音乐定制制作（受托方模式）

| 序号 | 必查条款 | 风险等级 | 审查要点 |
|------|---------|---------|---------|
| D-1 | 工作成果要求 | 🔴 | 需求明确、变更需书面确认 |
| D-2 | 修改次数限制 | 🔴 | 限制免费修改次数、额外修改另收费 |
| D-3 | 付款条款 | 🔴 | 分期付款、预付款、验收后尾款 |
| D-4 | 验收标准 | 🔴 | 客观标准、沉默视为验收 |

| D-6 | 署名权 | 🔴 | 乙方保留署名权，不可剥夺 |
| D-7 | 甲方解除权限制 | 🔴 | 限制随意解除、已发生费用补偿 |

## 风险分级规则

| 等级 | 标记 | 正文处理 | 批注处理 |
|------|------|---------|---------|
| 高风险 | 🔴 | 模式A修改 / 模式B仅批注 | 修改理由 + 法律依据 |
| 中风险 | 🟡 | 模式A修改 / 模式B仅批注 | 修改理由 + 法律依据 |
| 低风险 | 🟢 | **不修改** | 仅在报告中列出建议 |
| 商务条款 | 💼 | **不修改** | 仅批注风险提示，需当事人确认 |

**核心原则**：仅对🔴🟡中高风险条款修改；商务条款（合同金额、服务内容等）只批注不修改。

## 六维自检机制

每次审查输出前必须通过以下六项自检：

| 序号 | 自检项 | 核心问题 |
|------|--------|---------|
| 1 | 立场自检 | 是否始终保持乙方视角？ |
| 2 | 法律依据自检 | 所有引用是否来自法律依据池？ |
| 3 | 场景一致性自检 | 是否使用了正确的场景审查思路？ |
| 4 | 质量自检 | 必查条款是否全覆盖？建议是否可操作？ |
| 5 | 审阅模式自检 | 审阅稿是否符合审查模式规范？ |
| 6 | 审阅稿核对自检 | 报告中每一条修改建议是否都在审阅稿中体现？ |

> 任何一项未通过，必须修正后重新自检，全部通过才可输出。

## 输出格式

### 审查摘要报告

```
一、合同基本信息
二、场景识别
三、风险概览（🔴🟡🟢💼分级）
四、审查详情（逐条：原合同表述 → 问题分析 → 法律依据 → 修改/批注内容）
五、修改建议汇总（核对清单，审阅稿逐条核对）
六、谈判策略提示（必争/可让步/实务建议）
七、审查自检声明
```

### Word 审阅稿/批注稿

| 项目 | 规范 |
|------|------|
| 修改方式 | 审阅模式（修订模式 / Track Changes） |
| 审阅人名称 | `legal assistant` |
| 输出格式 | `.docx`（Word 格式） |
| 文件命名 | 模式A：`[原合同名]_审查稿_YYYYMMDD.docx`；模式B：`[原合同名]_批注稿_YYYYMMDD.docx` |

批注格式示例：

- 模式A 高风险：`【🔴高风险】[修改理由]。法律依据：《XXX》第X条。`
- 模式A 中风险：`【🟡中风险】[修改理由]。法律依据：《XXX》第X条。`
- 商务条款：`【💼商务条款】[风险提示]。此条款需合同当事人确认，未做修改。`

## WPS COM 技术要点

生成 Word 审阅稿时，WPS COM 的 `Find/Replace` 不产生修订记录，必须使用 **Delete + TypeText** 模拟用户输入：

```
正确流程（每条修改）:
1. Range.Find.Execute(FindText=原文) → 找到目标
2. Selection.SetRange → 选中
3. doc.Comments.Add → 添加批注
4. Selection.Delete() → 删除原文（产生修订删除标记）
5. Selection.TypeText(新文) → 键入新文（产生修订插入标记）
```

> 使用 `Range.Find`（从文档开头搜索）而非 `Selection.Find`（从上次位置开始），避免位置偏移导致搜索失败。

## 注意事项

1. **角色锁定**：本工具永远从乙方视角审查，不切换立场
2. **审查前必问**：每次审查开始前必须确认审查模式，未确认前不得修改合同正文
3. **先报告后审阅稿**：必须先输出审查摘要报告，再以报告为清单逐条生成审阅稿
4. **法律依据不编造**：所有法律引用必须来自 `modules/_legal-basis.md`，池外引用标注"待核实"
5. **场景识别不明确时追问**：不强行归类，主动向用户追问关键问题
6. **WPS COM 依赖**：生成 Word 审阅稿/批注稿需调用 WPS COM 接口，要求 Windows 环境 + 已安装 WPS + pywin32；仅输出审查摘要报告则无此依赖
7. **平台无关**：指令集以 Markdown 文件交付，任何支持读取结构化指令的 AI Agent 均可使用
